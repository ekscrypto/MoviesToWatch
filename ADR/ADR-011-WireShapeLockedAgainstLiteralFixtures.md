## Wire-shape contracts are locked against literal fixtures

Decision: Every Swift type whose `Codable` output is written to disk for later off-line consumption (the persisted movies list, any archive manifests, exported CSV) or sent over the wire to an external service ships two test artefacts beyond the auto-synthesized round-trip:

1. **Literal-fixture lock.** A test that encodes a fully-populated sample with `JSONEncoder` configured `[.prettyPrinted, .sortedKeys]` (or the format-appropriate equivalent — header line for CSV, deterministic ordering for any other text format) and asserts `String` equality against a literal payload pinned in the test file. The fixture covers every top-level key, every enum-discriminator case the type can emit, and every `Optional` in both its `nil` and non-`nil` shape.
2. **Schema-version pin and forward-rejection.** Types carrying a `schemaVersion` field expose a `currentSchemaVersion` constant, a test that pins it, and a custom `init(from:)` that throws when the decoded `schemaVersion` exceeds the current value. The throw is itself pinned by a test that decodes a payload at `currentSchemaVersion + 1`.

Round-trip equality alone (`encode(x) → decode → == x`) does not satisfy this ADR. A round-trip against the auto-synthesized `Codable` surface is self-validating: rename every `CodingKey`, flip every `Optional`'s encoding strategy, swap two adjacent fields — the round-trip still passes because it never compares against an external anchor. The literal fixture is that anchor.

The invariant binds `Codable` types written to disk or sent across a process boundary. It does not bind purely in-memory `Codable` shapes that never leave the process (e.g. test scaffolding encoded for diff-readable comparison); round-trip coverage is sufficient for those. New producers ship the schema-version guard from day one; legacy producers without one adopt it in the slice that first changes the schema.

Reasoning: On-disk persistence is a contract with the future version of the same app, and any wire payload is a contract with another process. A `CodingKey` typo that renames `releaseYear` to `releaseyear` in the encoded JSON ships silently under round-trip-only coverage; the encoder and decoder agree on the typo, every call site keeps compiling, and the regression only surfaces when a previously-saved file fails to decode on the next launch. That failure mode is the exact class the fluent suite (ADR-002, ADR-003) cannot reach — scenarios assert through `ViewRep`, which the typo doesn't disturb. The literal fixture is the smallest possible test an external consumer would have written; locking it in the test file replays that consumer's discipline inside the project. The schema-version pin is the asymmetric companion: a producer that bumps `currentSchemaVersion` without bumping the fixture has shipped an undocumented wire change.

## Review Scope

**Strengthens** findings about (a) `Codable` types written to disk or sent across a process boundary whose only test coverage is a round-trip pair, (b) PRs that add a new top-level key, rename a `CodingKey`, change an enum's encoded discriminator, or flip an `Optional`'s encoding strategy without updating the matching fixture in the same PR, (c) types carrying a `schemaVersion` field that have no `currentSchemaVersion` constant, no `init(from:)` rejection of higher versions, or no test pinning the constant, or (d) PRs that bump `currentSchemaVersion` without also bumping the literal fixture and the forward-rejection test.

**Drops** findings that ask for fixture-locking on purely in-memory `Codable` shapes that never leave the process, or that propose replacing the fixture with a programmatic equality check against another freshly-encoded sample (that's the round-trip — what this ADR exists to supplement, not duplicate).
