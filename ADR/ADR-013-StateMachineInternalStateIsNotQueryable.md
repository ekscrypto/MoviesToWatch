## StateMachine internal state is not directly queryable

Decision: `PersistentState` and `EphemeralState` are read in exactly two places: `Intent.*.mutate(persistent:ephemeral:)` (the serial mutation point ADR-001 defines) and `ViewRep` generation. The `StateMachine` actor MUST NOT expose `get…` / `read…` / `current…` accessors that hand internal state — or values derived from it — back to a caller, regardless of whether the caller is an `Activity`, an `Adapter`, the UI, or a test. Activities that need a value out of the StateMachine dispatch an `Intent` whose `mutate(...)` performs the read inside the serial point and writes the result into a reference-type carrier the activity already holds, or returns the value through an activity scheduled by `mutate(...)` (the pattern used by `ActivityPersistMovies`, which reads the freshest snapshot inside the actor via the `internal` helper `snapshotMovies()`). Tests observe via `StateMachine.viewReps` (ADR-003 / ADR-008); they do not gain a side door because they happen to live in the same module.

Reasoning: ADR-001's serial-mutation guarantee is what removes locking from the rest of the codebase and what makes race conditions reproducible by replaying the `Intent` stream against a captured prior state. A `getFoo(id:)` accessor on the StateMachine breaks both: the value it returns is stale the moment the actor hop completes (any `Intent` queued behind the read may have already changed the field), and the call site is a state-dependent branch that does not appear in the `Intent` stream, so the same captured stream replayed against the same prior state can produce different outcomes depending on how the actor scheduler interleaved the get with the next `Intent`. The `Intent`-as-carrier pattern keeps the read inside the serial point: the value the activity sees was the value at the instant that `Intent` ran, every read is reproducible from the `Intent` stream alone, and the activity has no way to branch on state without that branch being visible as an `Intent`.

## Review Scope

**Strengthens** findings about (a) new or existing `func get…` / `func read…` / `func current…` (or property accessors with the same effect) on `StateMachine` that return `PersistentState` / `EphemeralState` fields or values derived from them to non-`Intent` callers, (b) activities that branch on a StateMachine accessor instead of dispatching an `Intent` that performs the read inside `mutate(...)`, (c) tests that reach for a StateMachine accessor instead of asserting against the `viewReps` stream, or (d) Adapters that close over the StateMachine to read state out of band.

**Drops** findings that ask `Intent.*.mutate(...)` or `ViewRep` generation to stop reading `PersistentState` / `EphemeralState` directly — those two sites are the serial mutation point and its pure-function projection, and the invariant is precisely that they are the only readers.
